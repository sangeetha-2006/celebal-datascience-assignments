01 · System Diagram

How a question becomes an answer

Every message passes through four knowledge sources before a response is generated — the agent decides which ones actually matter for the query in front of it.

02 · Build Sequence

Seven phases, each one runnable on its own

The system was built incrementally — every phase works standalone via its own CLI, and the API in Phase 7 ties them all together.

1

Data Ingestion

Scrape → clean → chunk web content into retrievable pieces.

2

RAG

Embed chunks locally (sentence-transformers), retrieve via FAISS vector search.

3

Knowledge Graph

Extract entities/relationships with spaCy, store as a queryable graph in Neo4j.

4

Long-Term Memory

Persist user preferences and full conversation history in SQLite.

5

LangGraph Agent

Orchestrate RAG + KG + memory + live tools, generate with a local LLM.

6

Evaluation

Score context relevance, answer relevance, and faithfulness — no manual grading.

7

API Layer

Expose everything through FastAPI, with this interface as the frontend.

WHY NOT PERFECT

A small local LLM occasionally blends retrieved facts with its own training knowledge instead of staying strictly grounded — the tradeoff for running fully free and local, regardless of model size.

DESIGN CHOICE

SQLite replaces MongoDB/Postgres, spaCy replaces LLM-based extraction, and Ollama replaces a hosted API — each swap trades some accuracy for zero cost and zero setup friction.
